PART 1

Foundations

01What Angular Is

Angular is a full, opinionated SPA framework (not just a library) built on TypeScript. It ships routing, HTTP, forms, dependency injection, and testing out of the box — "batteries included," versus React's assemble-your-own approach.

Angular vs AngularJS

"Angular" means version 2+ (a complete rewrite in TypeScript with components). "AngularJS" is the legacy 1.x. Don't conflate them in an interview.

02Architecture & the CLI

An Angular app is a tree of components, backed by services injected via DI, organised (classically) into modules. The Angular CLI scaffolds, builds, serves, and tests.

bash
ng new my-app          # scaffold a project
ng generate component user   # or: ng g c user
ng generate service api
ng serve               # dev server with live reload
ng build               # production bundle

03NgModules vs Standalone Components

Historically every app had a root NgModule (AppModule) declaring components and importing feature modules. Since Angular 14 (default in 17+), standalone components remove the boilerplate — a component declares its own imports.

typescript
@Component({
  selector: 'app-user',
  standalone: true,
  imports: [CommonModule, RouterLink],   // no NgModule needed
  templateUrl: './user.component.html'
})
export class UserComponent {}
Say this

"Standalone components are the modern default — they cut the NgModule ceremony and make lazy loading and testing simpler. Older codebases still use NgModules."

PART 2

Components & Templates

04Anatomy of a Component

A component = a class (state + logic) + a template (HTML with bindings) + styles, tied together by the @Component decorator.

typescript
@Component({
  selector: 'app-counter',
  template: `<button (click)="inc()">{{ count }}</button>`,
  styles: [`button { padding: 8px; }`]
})
export class CounterComponent {
  count = 0;
  inc() { this.count++; }
}

05Lifecycle Hooks

HookRuns when
ngOnChangesAn @Input value changes (before ngOnInit).
ngOnInitOnce, after first inputs set — do initial setup / data fetch here.
ngAfterViewInitAfter the component's view (and ViewChildren) initialise.
ngOnDestroyBefore removal — unsubscribe, clean up timers.
Why not the constructor?

Use the constructor only for DI. Do data fetching in ngOnInit — by then inputs are bound and the component is ready.

07Directives & Control Flow

Three kinds of directives: components (directives with a template), structural (change the DOM), and attribute (change appearance/behaviour).

html
<!-- classic structural directives -->
<p *ngIf="user">{{ user.name }}</p>
<li *ngFor="let item of items; trackBy: trackById">{{ item.name }}</li>

<!-- new built-in control flow (Angular 17+) -->
@if (user) { <p>{{ user.name }}</p> } @else { <p>Guest</p> }
@for (item of items; track item.id) { <li>{{ item.name }}</li> }

<!-- attribute directives -->
<div [ngClass]="{ active: isActive }" [ngStyle]="{ color: c }"></div>
trackBy matters

Without trackBy/track, *ngFor re-renders the whole list on change. Tracking by id lets Angular reuse DOM nodes — a common performance question.

PART 3

Component Communication

08@Input & @Output

Parent → child via @Input; child → parent via @Output + EventEmitter.

typescript
// child.component.ts
@Input() user!: User;                 // receive data
@Output() selected = new EventEmitter<User>();  // emit up
choose() { this.selected.emit(this.user); }
html
<!-- parent template -->
<app-child [user]="current" (selected)="onSelect($event)"></app-child>

For unrelated components, share state through a service holding an observable (e.g. a BehaviorSubject) both subscribe to.

09Template Refs & ViewChild

typescript
// grab a child component or DOM element from the template
@ViewChild(ChildComponent) child!: ChildComponent;
@ViewChild('nameInput') input!: ElementRef;   // <input #nameInput>

ngAfterViewInit() { this.input.nativeElement.focus(); }
Timing

ViewChild references are available from ngAfterViewInit onward, not in ngOnInit.

PART 4

Services, DI & Routing

10Services & Dependency Injection

Put reusable logic and state in services; Angular's DI provides them where needed. providedIn: 'root' registers a single app-wide (singleton) instance and is tree-shakable.

typescript
@Injectable({ providedIn: 'root' })
export class UserService {
  constructor(private http: HttpClient) {}
  getUsers() { return this.http.get<User[]>('/api/users'); }
}

// inject via constructor (or the inject() function)
constructor(private users: UserService) {}
DI hierarchy

Angular has a hierarchical injector: a service provided in root is a singleton; provided in a component, each instance of that component gets its own. This scoping is a favourite question.

11Routing, Guards & Lazy Loading

typescript
const routes: Routes = [
  { path: '', component: HomeComponent },
  { path: 'users/:id', component: UserComponent, canActivate: [authGuard] },
  { path: 'admin',
    loadComponent: () => import('./admin.component').then(m => m.AdminComponent) } // lazy
];
  • <router-outlet> renders the matched route; routerLink navigates without a full reload.
  • Read params via ActivatedRoute (route.paramMap).
  • Guards (canActivate, canDeactivate) protect routes; resolvers pre-fetch data before activation.
  • Lazy loading splits code per route so the initial bundle stays small.
PART 5

Data & RxJS

12HttpClient

HttpClient returns Observables, not Promises. Nothing happens until you subscribe — or let the async pipe subscribe for you.

typescript
users$ = this.userService.getUsers();   // Observable<User[]>
html
<!-- async pipe subscribes AND unsubscribes automatically -->
<li *ngFor="let u of users$ | async">{{ u.name }}</li>
Prefer the async pipe

It handles subscribe/unsubscribe for you, preventing memory leaks. Manual .subscribe() must be cleaned up in ngOnDestroy.

13RxJS & Operators

RxJS models async data as streams (Observables) you transform with operators inside a pipe().

typescript
this.search.valueChanges.pipe(
  debounceTime(300),                // wait for typing to pause
  distinctUntilChanged(),           // ignore unchanged input
  switchMap(q => this.api.search(q)), // cancel prior request, run new one
  catchError(() => of([]))          // recover from errors
).subscribe(results => this.results = results);
OperatorUse
mapTransform each value.
switchMapSwitch to a new inner stream, cancelling the previous (search/type-ahead).
mergeMap / concatMapRun inner streams in parallel / in order.
catchErrorHandle/recover from errors.
takeUntilAuto-unsubscribe pattern.
Subject vs BehaviorSubject

A Subject is a multicast Observable; a BehaviorSubject also holds and emits the latest value to new subscribers — ideal for shared state in a service.

14HTTP Interceptors

Interceptors sit in the HTTP pipeline to transform every request/response — perfect for auth tokens, logging, and global error handling.

typescript
export const authInterceptor: HttpInterceptorFn = (req, next) => {
  const token = inject(AuthService).token;
  return next(req.clone({
    setHeaders: { Authorization: `Bearer ${token}` }
  }));
};

16Change Detection

Angular uses Zone.js to detect async events (clicks, timers, HTTP) and re-check the component tree to update the DOM. That default checks everything on each cycle.

OnPush

ChangeDetectionStrategy.OnPush tells Angular to re-check a component only when its @Input reference changes, an event fires in it, or an observable it uses (via async pipe) emits. Combined with immutable data, it dramatically cuts work — a top performance answer.

17Signals (Angular 16+)

Signals are a reactive primitive: a wrapped value that notifies consumers when it changes, enabling fine-grained updates without relying on Zone.js.

typescript
count = signal(0);
double = computed(() => this.count() * 2);  // derived, auto-updates

inc() { this.count.update(n => n + 1); }        // read with count(), set/update to change
Direction of travel

Signals are Angular's future reactivity model — more granular and predictable than Zone-based change detection. Mention them to show you track current Angular.

18Pipes

Pipes transform values in the template: {{ value | pipe:args }}. Built-ins include date, currency, uppercase, json, async.

html
{{ price | currency:'INR' }}   {{ createdAt | date:'short' }}
Pure vs impure

Pure pipes (default) re-run only when the input reference changes — efficient. Impure pipes run every cycle; use sparingly.

PART 7

Quick-Fire Q&A

Tap a question to reveal the answer.

>Angular vs AngularJS?

Angular (2+) is a TypeScript, component-based rewrite; AngularJS (1.x) is the legacy scope/controller framework. They are not compatible.

>What is a component?

A class with a template and styles, marked by @Component — the building block of the UI tree.

>Four types of data binding?

Interpolation {{ }}, property [ ], event ( ), and two-way [( )].

>Where should you fetch data, and why?

In ngOnInit — inputs are bound and the component is initialised; the constructor is only for DI.

>How do components communicate?

Parent-to-child via @Input, child-to-parent via @Output/EventEmitter, and unrelated components via a shared service (often a BehaviorSubject).

>providedIn: 'root' — what does it mean?

The service is a tree-shakable, app-wide singleton in the root injector.

>Observable vs Promise?

Observables are lazy, can emit multiple values over time, and are cancellable with rich operators; Promises are eager and resolve once.

>switchMap vs mergeMap?

switchMap cancels the previous inner observable when a new value arrives (ideal for search); mergeMap runs them concurrently.

>How do you prevent memory leaks from subscriptions?

Use the async pipe, or unsubscribe in ngOnDestroy (e.g. takeUntil a destroy subject).

>What does OnPush change detection do?

Limits re-checks to input reference changes, component events, or async-pipe emissions — big performance win with immutable data.

>What are Signals?

A reactive primitive (value + change notifications) enabling fine-grained updates without Zone.js; Angular's emerging reactivity model.

>What is lazy loading?

Loading a route's code only when navigated to, keeping the initial bundle small.

Final tips for the interview

  1. Explain the component/service/DI architecture as the core mental model.
  2. Know the four bindings and when ngOnInit vs the constructor runs.
  3. Be fluent in RxJS: observables, switchMap/mergeMap, and the async pipe.
  4. Always mention unsubscribing to avoid memory leaks.
  5. Cite OnPush, trackBy, and lazy loading for performance.
  6. Show current knowledge: standalone components and signals.
