PART 1

Foundations

01Overview

MySQL is the most widely used open-source RDBMS, popular in web stacks (the "M" in LAMP). It is fast, easy to run, and pairs a SQL layer with pluggable storage engines.

02Storage Engines: InnoDB vs MyISAM

InnoDB (default)MyISAM (legacy)
TransactionsYes (ACID)No
LockingRow-levelTable-level
Foreign keysYesNo
Crash recoveryYes (redo log)No
Best forAlmost everythingRare read-only/legacy
Default answer

"Use InnoDB." It gives transactions, row-level locking, foreign keys and crash recovery. MyISAM is legacy — only for niche read-only cases.

SQL & Features

04MySQL SQL Features

sql
SELECT * FROM orders ORDER BY total DESC LIMIT 10 OFFSET 20;  -- paging
SELECT IFNULL(commission, 0);                  -- NULL handling
SELECT customer_id, GROUP_CONCAT(product) FROM items GROUP BY customer_id;
LIMIT/OFFSET paging

Simple but slow at large offsets (it still scans skipped rows). For deep pagination, prefer keyset/seek pagination (WHERE id > :last ORDER BY id LIMIT n).

05JSON & Upsert

sql
-- upsert: insert, or update on duplicate key
INSERT INTO counters(id, hits) VALUES (1, 1)
ON DUPLICATE KEY UPDATE hits = hits + 1;

-- query a JSON column
SELECT data->>'$.email' FROM users WHERE data->>'$.active' = 'true';
PART 3

Indexes & Performance

06Indexes

InnoDB uses a clustered index: the primary key is the row storage (data is stored in PK order). Secondary indexes store the PK value and look up the row via it.

  • Keep the PK small and monotonic (e.g. auto-increment) — random PKs cause page splits and fragmentation.
  • Composite index follows the leftmost-prefix rule: an index on (a,b,c) helps a, a,b, a,b,c — not b alone.
  • Covering index answers a query entirely from the index, skipping the row lookup.

07EXPLAIN & Tuning

sql
EXPLAIN SELECT * FROM orders WHERE customer_id = 42;

Read the type column: const/eq_ref/ref use indexes well; ALL means a full table scan. Check key (index chosen) and rows (estimated scanned). Enable the slow query log to find offenders in production.

PART 4

Transactions & Concurrency

08Transactions & MVCC

InnoDB is fully ACID and uses MVCC (multi-version concurrency control): reads see a consistent snapshot without blocking writers, using undo logs to reconstruct old row versions.

sql
START TRANSACTION;
UPDATE accounts SET bal = bal - 100 WHERE id = 1;
UPDATE accounts SET bal = bal + 100 WHERE id = 2;
COMMIT;   -- or ROLLBACK

09Locks & Isolation

MySQL InnoDB's default isolation is REPEATABLE READ (unusual — most DBs default to READ COMMITTED). It uses gap locks and next-key locks to prevent phantom reads at that level.

Deadlocks

InnoDB detects deadlocks and rolls back the smaller transaction; your app should catch the error and retry. Access rows in a consistent order and keep transactions short to avoid them.

PART 5

Scaling & Ops

10Replication & High Availability

MySQL replicates by shipping the binary log (binlog) from a primary to replicas.

  • Async replication (default) — primary doesn't wait; replicas may lag.
  • Semi-sync — primary waits for at least one replica to acknowledge.
  • Common uses: read replicas to scale reads, and failover for HA.
Read-after-write

Because replicas can lag, reading your own write from a replica may return stale data — route critical reads to the primary or use a consistency strategy.

PART 6

Quick-Fire Q&A

Tap a question to reveal the answer.

>InnoDB vs MyISAM?

InnoDB is transactional, row-locking, supports foreign keys and crash recovery — the default and right choice. MyISAM is legacy, table-locking, no transactions.

>What is MySQL's default isolation level?

REPEATABLE READ (InnoDB), which also prevents phantom reads via gap/next-key locks — unusual, since most databases default to READ COMMITTED.

>How does InnoDB store rows?

As a clustered index on the primary key — the data is stored in PK order, and secondary indexes reference the PK.

>Why keep the PK small and monotonic?

Because it is the clustered index; large or random PKs cause page splits, fragmentation, and bloated secondary indexes.

>What is the leftmost-prefix rule?

A composite index on (a,b,c) can serve queries filtering a, or a+b, or a+b+c — but not b or c alone.

>How does MySQL scale reads?

Binlog-based replication to read replicas; route reads to replicas and writes to the primary, mindful of replication lag.

>Problem with LIMIT/OFFSET pagination?

Large offsets still scan and discard skipped rows; use keyset (seek) pagination for deep pages.

>How do you do an upsert?

INSERT ... ON DUPLICATE KEY UPDATE, which inserts or updates when a unique/PK conflict occurs.

>What is MVCC?

Multi-version concurrency control — readers see a consistent snapshot via undo logs without blocking writers.

>DECIMAL vs FLOAT for money?

Use DECIMAL for exact values; FLOAT/DOUBLE are approximate and cause rounding errors.

Final tips for the interview

  1. Lead with InnoDB and its transactional, row-locking properties.
  2. Know MySQL's REPEATABLE READ default and gap locks.
  3. Explain the clustered PK and why PK choice matters.
  4. Apply the leftmost-prefix rule and covering indexes.
  5. Read EXPLAIN output and use the slow query log.
  6. Discuss read replicas and replication lag for scaling.
