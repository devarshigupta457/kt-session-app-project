PART 1

Fundamentals

01What Git Is

Git is a distributed version-control system: every clone is a full copy of the history, so you can commit, branch, and view history offline. GitHub/GitLab are hosting services on top of Git — not Git itself.

Snapshots, not diffs

Git stores each commit as a snapshot of the whole project (unchanged files are referenced, not re-stored), each identified by a SHA hash.

02The Three Areas

Working Directory your files, edited | git add v Staging Area (index) what goes in the next commit | git commit v Local Repository committed history | git push v Remote shared (origin)
Say this

"add stages, commit records to local history, push shares to the remote." HEAD is a pointer to your current commit/branch.

03Starting a Repo

bash
git init                 # new repo in current folder
git clone <url>          # copy an existing remote repo
git config --global user.name  "Your Name"
git config --global user.email "you@example.com"
PART 2

Everyday Git

04Daily Commands

bash
git status               # what's changed / staged
git add file.js          # stage one file
git add .                # stage everything
git commit -m "msg"    # record staged changes
git commit -am "msg"   # stage tracked + commit in one
git diff                 # unstaged changes
git diff --staged        # staged changes
Good commits

Small, focused commits with clear imperative messages ("Add login validation") make history reviewable and easy to revert.

05.gitignore

List files/patterns Git should never track (build output, secrets, dependencies).

gitignore
node_modules/
target/
*.log
.env            # never commit secrets
Already tracked?

.gitignore only affects untracked files. To stop tracking something already committed: git rm --cached <file>.

06Inspecting History

bash
git log --oneline --graph   # compact, visual history
git show <commit>            # details of one commit
git blame file.js           # who last changed each line
PART 3

Branching & Merging

07Branches

A branch is a movable pointer to a commit — cheap and fast in Git. Work on features in isolation, then merge.

bash
git branch                   # list branches
git switch -c feature/login  # create + switch (modern)
git checkout -b feature/login # older equivalent
git switch main              # switch back

08Merge vs Rebase

MergeRebase
HistoryPreserves branches; adds a merge commitRewrites — replays commits onto the base for a linear history
SafetyNon-destructiveRewrites hashes — don't rebase shared/pushed branches
bash
git merge feature      # bring feature into current branch
git rebase main        # replay my commits on top of main
Golden rule

Merge to combine finished work; rebase to tidy local history before sharing. Never rebase commits others have pulled.

09Resolving Conflicts

A conflict happens when two branches change the same lines. Git marks them; you edit, then stage and continue.

text
<<<<<<< HEAD
your change
=======
their change
>>>>>>> feature

Fix the file, remove the markers, then git add and git commit (or git rebase --continue).

PART 4

Remotes & Collaboration

10Remotes

bash
git remote -v            # list remotes (usually 'origin')
git fetch                # download commits, DON'T merge
git pull                 # fetch + merge into current branch
git push                 # upload local commits
git push -u origin feat  # push + set upstream tracking
fetch vs pull

fetch only downloads — you review before integrating. pull = fetch + merge (or --rebase) in one step.

11Collaboration Flow

1 git switch -c feature/x # branch off main 2 ...commit work... 3 git push -u origin feature/x 4 open a Pull Request on GitHub # review + CI 5 address review, merge PR 6 git switch main && git pull # sync local main
Feature-branch workflow

Never commit straight to main. Branch per feature, open a PR, get review + passing CI, then merge. This is the standard team model.

PART 5

Undoing & Recovering

12Undoing Changes

CommandEffect
git restore fileDiscard unstaged changes to a file.
git reset --soft HEAD~1Undo last commit, keep changes staged.
git reset --mixed HEAD~1Undo commit, keep changes unstaged (default).
git reset --hard HEAD~1Undo commit & discard changes (destructive).
git revert <commit>New commit that undoes an old one (safe for shared history).
git commit --amendFix the last commit's message or contents.
reset vs revert

reset rewrites history (use locally). revert adds a new inverse commit (use on shared branches).

13Stash & Recover

bash
git stash            # shelve uncommitted work
git stash pop        # bring it back
git cherry-pick <sha> # apply one commit from elsewhere
git reflog           # log of where HEAD has been
Lifesaver

git reflog tracks every move of HEAD — even after a bad reset --hard you can usually find the lost commit's SHA and restore it.

PART 6

Quick-Fire Q&A

Tap a question to reveal the answer.

>Git vs GitHub?

Git is the distributed version-control tool that runs locally; GitHub is a hosting platform (remotes, PRs, CI) built on top of Git.

>What are the three areas?

Working directory (edits), staging area/index (next commit), and the local repository (committed history) — plus the remote.

>git fetch vs git pull?

fetch downloads remote commits without changing your working branch; pull is fetch + merge (or rebase).

>Merge vs rebase?

Merge preserves history and adds a merge commit; rebase replays commits for a linear history but rewrites hashes — never rebase shared branches.

>reset vs revert?

reset moves the branch pointer and can discard commits (rewrites history, local only); revert creates a new commit that undoes a previous one (safe for shared history).

>Difference between reset --soft, --mixed, --hard?

Soft keeps changes staged, mixed (default) keeps them unstaged, hard discards them entirely.

>How do you resolve a merge conflict?

Open the marked files, choose/combine the correct code, remove the conflict markers, then git add and commit (or continue the rebase).

>What is HEAD?

A pointer to your current commit, usually via the current branch. HEAD~1 is its parent.

>How do you undo a pushed commit safely?

Use git revert to add an inverse commit, so shared history stays intact rather than rewriting it.

>What is git stash for?

Temporarily shelving uncommitted changes so you can switch context (e.g. fix a bug on another branch), then restore them with git stash pop.

Final tips for the interview

  1. Draw the three areas and the add / commit / push flow — the core mental model.
  2. Explain merge vs rebase and the golden rule about shared branches.
  3. Be exact on reset vs revert and the three reset modes.
  4. Describe the feature-branch + PR workflow from real experience.
  5. Know how to resolve conflicts step by step.
  6. Mention reflog as your safety net — it signals real fluency.
