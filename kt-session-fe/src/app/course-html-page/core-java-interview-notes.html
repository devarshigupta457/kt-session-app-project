PART 1

Language & OOP Fundamentals

01JVM, JRE, JDK & Bytecode

Java's headline feature is "write once, run anywhere." Your .java source is compiled by javac into platform-neutral bytecode (.class), which the JVM interprets/JIT-compiles for whatever OS it runs on.

TermWhat it is
JVMJava Virtual Machine — runs bytecode, manages memory & garbage collection. Platform-specific.
JREJava Runtime Environment — JVM + core libraries. Enough to run Java apps.
JDKJava Development Kit — JRE + dev tools (javac, jar, debugger). Enough to build apps.
One-liner

JDK ⊇ JRE ⊇ JVM. The JIT (Just-In-Time) compiler inside the JVM converts hot bytecode to native machine code at runtime for speed.

    @Override String sound() { return "Woof"; }   // overriding

}

06static, final, this, super

  • static — belongs to the class, not an instance. Shared across all objects; accessed via the class name. Used for constants, utility methods, counters.
  • final — variable: can't be reassigned; method: can't be overridden; class: can't be extended (e.g. String).
  • this — reference to the current object (disambiguate fields, chain constructors with this(...)).
  • super — reference to the parent (call super() constructor or super.method()).
Gotcha

final on a reference means the reference can't change, but the object it points to can still mutate — final List<X> l can still have items added.

PART 2

Types, Strings & Memory

07Primitives, Wrappers & Autoboxing

Java has 8 primitives (byte short int long float double char boolean) stored by value, plus wrapper objects (Integer, Long…) for use in generics/collections.

java
Integer boxed = 5;      // autoboxing: int -> Integer
int back = boxed;        // unboxing: Integer -> int
List<Integer> nums = new ArrayList<>(); // collections need objects
Cache gotcha

Integer caches values −128..127, so Integer.valueOf(100) == valueOf(100) is true but 200 == 200 (as Integers) is false. Always compare wrappers with .equals().

08String, the Pool & Immutability

String is immutable — every "change" creates a new object. String literals live in the String pool and are reused.

java
String a = "hi";              // pooled
String b = "hi";              // same pooled object -> a == b is true
String c = new String("hi");  // new heap object -> a == c is false

StringBuilder vs StringBuffer

StringStringBuilderStringBuffer
MutableNoYesYes
Thread-safeYes (immutable)NoYes (synchronized)
Use whenFixed textHeavy edits, single threadHeavy edits, shared thread
Why immutable matters

Immutability makes String safe to share and cache, thread-safe by default, and usable as a reliable HashMap key.

09Stack, Heap & Garbage Collection

STACK (per thread) HEAP (shared) +--------------------++---------------------+ | main() frame | | new User(...) | | int x = 5 | | new int[1000] | | User u ----------+|-------->| (objects live here) | +--------------------+ +---------------------+ values & references objects & instance data
  • Stack — one per thread; holds method frames, local primitives, and object references. Freed automatically when a method returns.
  • Heap — shared; holds all objects. Cleaned by the garbage collector.
Garbage collection

An object is eligible for GC when nothing reachable references it. Modern GCs (G1, ZGC) use generations — most objects die young in the Young generation; survivors are promoted to Old. You never free() manually.

10== vs equals() and the hashCode Contract

  • == compares references for objects (or raw values for primitives).
  • equals() compares logical equality (you override it for value semantics).
The contract

If you override equals(), you must override hashCode(). Equal objects must return equal hash codes — otherwise they get lost in HashMap/HashSet.

java
@Override public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof User u)) return false;
    return id == u.id && Objects.equals(name, u.name);
}
@Override public int hashCode() { return Objects.hash(id, name); }

11Java is Always Pass-by-Value

Java always passes arguments by value. For objects, the value of the reference is copied — so you can mutate the object, but reassigning the parameter doesn't affect the caller.

java
void tweak(StringBuilder sb, int n) {
    sb.append("!");   // caller SEES this (same object)
    sb = null;        // caller does NOT see this (copy of reference)
    n = 99;           // caller does NOT see this (primitive copy)
}
PART 3

Collections & Generics

12Collections Hierarchy

Iterable └─ Collection ├─ List → ArrayList, LinkedList, Vector (ordered, duplicates) ├─ Set → HashSet, LinkedHashSet, TreeSet (no duplicates) └─ Queue → LinkedList, PriorityQueue, ArrayDeque Map (separate root) → HashMap, LinkedHashMap, TreeMap, Hashtable (key → value)
Quick picks

ArrayList for indexed access · LinkedHashSet for unique + insertion order · TreeMap for sorted keys · ArrayDeque for stacks/queues.

13Lists & Maps — Internals

ArrayList vs LinkedList

HashMap internals

  • Array of buckets; index = hash(key) & (capacity-1).
  • Collisions chain in a linked list, which converts to a balanced tree once a bucket exceeds 8 entries (Java 8+) for O(log n) lookups.
  • Default capacity 16, load factor 0.75 — it resizes (doubles) when 75% full.

// natural order via Comparable
class User implements Comparable<User> {
    public int compareTo(User o) { return Integer.compare(age, o.age); }
}
// custom orders via Comparator (Java 8 style)
users.sort(Comparator.comparing(User::getName)
                     .thenComparing(User::getAge).reversed());

15Generics & Wildcards

Generics give compile-time type safety and remove casts. At runtime the type is erased (type erasure) — the JVM sees raw types.

java
List<? extends Number> producer;  // read from it (upper bound)
List<? super Integer>  consumer;  // write to it (lower bound)
PECS

Producer Extends, Consumer Super. Use ? extends T when a structure produces values you read; ? super T when it consumes values you write.

16Fail-Fast vs Fail-Safe Iterators

  • Fail-fast (ArrayList, HashMap): throw ConcurrentModificationException if the collection is structurally modified during iteration. They track a modCount.
  • Fail-safe (CopyOnWriteArrayList, ConcurrentHashMap): iterate over a copy/snapshot, so concurrent edits don't throw — but you may not see the latest data.
Safe removal

To remove while iterating a fail-fast list, use the iterator's own iterator.remove() or removeIf(...).

PART 4

Exceptions & Concurrency

17Exception Handling

Throwable ├─ Error → JVM problems (OutOfMemoryError) — don't catch └─ Exception ├─ checked → IOException, SQLException — compiler forces handling └─ RuntimeException (unchecked) → NPE, IllegalArgument, ...
  • Checked: must be caught or declared with throws. Recoverable, external conditions.
  • Unchecked (RuntimeException): programming bugs; not forced by the compiler.
  • throw raises an exception; throws declares one in a signature.
  • finally always runs (except System.exit() / JVM crash) — used for cleanup.
java
// try-with-resources: auto-closes anything AutoCloseable
try (BufferedReader r = Files.newBufferedReader(path)) {
    return r.readLine();
} catch (IOException e) {
    throw new UncheckedIOException(e);
}   // r.close() called automatically, even on exception

18Threads & Lifecycle

Create a thread by implementing Runnable (preferred) or Callable (returns a value), rather than extending Thread.

NEW --start()--> RUNNABLE <--> RUNNING --run() ends--> TERMINATED | wait/lock/sleep -> BLOCKED / WAITING / TIMED_WAITING
java
Runnable task = () -> System.out.println("run on " +
        Thread.currentThread().getName());
new Thread(task).start();   // start() spawns a thread; run() would NOT
Classic trap

Calling run() directly executes on the current thread. Only start() creates a new one.

19synchronized, volatile, wait/notify

ToolGuaranteesNote
synchronizedMutual exclusion + visibilityOne thread in the block at a time (intrinsic lock).
volatileVisibility onlyReads/writes hit main memory; NOT atomic for i++.
wait()/notify()Thread coordinationCalled inside synchronized on the same monitor.
Visibility vs atomicity

volatile fixes visibility (threads see the latest value) but not atomicity. For counters use AtomicInteger or synchronized.

20Executors & Concurrency Utilities

Prefer the Executor framework over raw threads — it pools and reuses threads instead of creating one per task.

java
ExecutorService pool = Executors.newFixedThreadPool(4);
Future<Integer> f = pool.submit(() -> expensiveCalc()); // Callable
Integer result = f.get();   // blocks until done
pool.shutdown();

Useful utilities (java.util.concurrent)

  • AtomicInteger/AtomicLong — lock-free counters via CAS (compare-and-swap).
  • ReentrantLock — explicit lock with tryLock(), fairness, conditions.
  • ConcurrentHashMap, CopyOnWriteArrayList — thread-safe collections.
  • CountDownLatch, CompletableFuture — coordinate and compose async work.
PART 5

Modern Java (8+)

21Lambdas & Functional Interfaces

A functional interface has exactly one abstract method (@FunctionalInterface). A lambda is a concise implementation of it.

java
Runnable r = () -> System.out.println("hi");
Function<String,Integer> len = String::length;   // method reference
BiFunction<Integer,Integer,Integer> add = (a,b) -> a + b;
Built-in interfaces

Supplier<T> (gives), Consumer<T> (takes), Function<T,R> (transforms), Predicate<T> (tests → boolean).

22The Stream API

Streams process collections declaratively. Intermediate ops (map, filter) are lazy; a terminal op (collect, forEach, reduce) triggers execution.

java
List<String> names = users.stream()
    .filter(u -> u.getAge() > 18)      // intermediate (lazy)
    .map(User::getName)               // intermediate (lazy)
    .sorted()
    .collect(Collectors.toList());    // terminal -> runs the pipeline

Map<String,List<User>> byCity = users.stream()
    .collect(Collectors.groupingBy(User::getCity));
Key point

Streams don't mutate the source and can't be reused after a terminal op. Use parallelStream() only for large, CPU-bound, stateless work.

23Optional & Friends

Optional<T> models "maybe a value" to avoid NullPointerException — use it as a return type, not for fields or parameters.

java
Optional<User> found = repo.findById(id);
String name = found.map(User::getName)
                   .orElse("Unknown");
found.ifPresent(u -> notify(u));   // avoid found.get() without a check

Other 8+ wins worth naming: the new Date/Time API (LocalDate, LocalDateTime, immutable & thread-safe), default methods on interfaces, and method references (Class::method).

PART 6

Quick-Fire Q&A

Tap a question to reveal the answer — try answering out loud first.

>Why is String immutable?

For security, thread-safety, caching (the string pool), and safe use as hash keys. Any "modification" returns a new String.

>Difference between == and equals()?

== compares references (or primitive values); equals() compares logical content when overridden.

>Why override hashCode() when you override equals()?

Hash-based collections use hashCode() to find the bucket, then equals() within it. If equal objects have different hash codes they'll be stored in different buckets and can't be found.

>Abstract class or interface — how do you choose?

Interface for a shared capability across unrelated types and when you need multiple inheritance of type. Abstract class when related classes share state and implementation.

>Is Java pass-by-value or pass-by-reference?

Always pass-by-value. For objects the value copied is the reference, so you can mutate the object but not reassign the caller's variable.

>How does HashMap work internally?

An array of buckets indexed by the key's hash. Collisions chain in a linked list that becomes a tree past 8 entries (Java 8+). It resizes at 75% load (default capacity 16).

>ArrayList vs LinkedList — when each?

ArrayList for fast random access and mostly-append workloads; LinkedList only when you add/remove at the ends a lot. In practice ArrayList wins most of the time.

>Checked vs unchecked exceptions?

Checked (compile-time enforced, e.g. IOException) for recoverable external conditions; unchecked (RuntimeException) for programming errors like NPE.

>volatile vs synchronized?

volatile guarantees visibility only; synchronized guarantees visibility + mutual exclusion. volatile alone can't make i++ atomic.

>What is type erasure?

Generics are a compile-time feature. The compiler erases type parameters, so at runtime List<String> and List<Integer> are both just List.

>Intermediate vs terminal stream operations?

Intermediate ops (map, filter) are lazy and return a stream; a terminal op (collect, reduce, forEach) triggers evaluation and produces a result.

>run() vs start() on a Thread?

start() creates a new thread and calls run() on it; calling run() directly just executes on the current thread — no concurrency.

Final tips for the interview

  1. Explain OOP with a real example from your code, not textbook definitions.
  2. Be ready to describe HashMap internals and the equals/hashCode contract.
  3. Know stack vs heap and roughly how garbage collection works.
  4. Distinguish volatile vs synchronized and visibility vs atomicity.
  5. Show comfort with Streams, lambdas, and Optional — expected at 3+ years.
  6. When unsure, reason about time/space complexity out loud.
