PART 1

Fundamentals

01DBMS vs RDBMS

A DBMS manages data storage and retrieval. An RDBMS is a DBMS that organises data into relations (tables) of rows and columns with enforced relationships and constraints, queried with SQL. Examples: Oracle, MySQL, PostgreSQL, SQL Server.

SQL vs NoSQL

Relational (SQL) databases favour structured data, strong consistency and joins; NoSQL (document, key-value, column, graph) trades some of that for flexible schemas and horizontal scale. Pick based on data shape and access patterns.

02The Relational Model

  • Table (relation) — a set of rows about one entity.
  • Row (tuple / record) — one instance.
  • Column (attribute) — one field, with a data type.
  • Schema — the structure/definition; instance = the data at a moment.
  • Constraints — rules the data must satisfy (NOT NULL, UNIQUE, CHECK, PK, FK).

03SQL Sub-Languages

CategoryPurposeCommands
DDLDefine structureCREATE ALTER DROP TRUNCATE
DMLManipulate dataSELECT INSERT UPDATE DELETE
DCLPermissionsGRANT REVOKE
TCLTransactionsCOMMIT ROLLBACK SAVEPOINT
DELETE vs TRUNCATE vs DROP

DELETE (DML) removes rows and can be rolled back and filtered; TRUNCATE (DDL) removes all rows fast, resets, and usually can't be rolled back; DROP removes the whole table. Classic question.

PART 2

Design & Modelling

04Keys

KeyMeaning
Primary keyUniquely identifies a row; NOT NULL + UNIQUE; one per table.
Candidate keyAny column(s) that could serve as the PK.
Foreign keyReferences a PK in another table; enforces referential integrity.
Unique keyEnforces uniqueness but allows one NULL (DB-dependent).
Composite keyA key made of multiple columns.

05Relationships

  • One-to-one — each row relates to at most one row in another table.
  • One-to-many — the common case (a customer has many orders): FK on the "many" side.
  • Many-to-many — modelled with a junction/bridge table holding two FKs (students ↔ courses via enrolments).

06Normalization

Normalization organises data to reduce redundancy and avoid update/insert/delete anomalies.

FormRule
1NFAtomic values; no repeating groups or arrays in a column.
2NF1NF + no partial dependency (non-key columns depend on the whole composite key).
3NF2NF + no transitive dependency (non-key columns depend only on the key).
BCNFStricter 3NF: every determinant is a candidate key.
Denormalization

Sometimes you deliberately duplicate data (denormalize) to avoid expensive joins and speed up reads — a tradeoff of write complexity for read performance, common in reporting/analytics.

PART 3

Querying

07SELECT & Logical Order of Execution

SQL is written in one order but evaluated in another — knowing this explains why you can't use a SELECT alias in WHERE.

written: SELECT ... FROM ... WHERE ... GROUP BY ... HAVING ... ORDER BY ... evaluated: 1 FROM / JOIN pick & combine tables 2 WHERE filter rows 3 GROUP BY form groups 4 HAVING filter groups 5 SELECT choose columns / aliases 6 ORDER BY sort 7 LIMIT take N

08Joins

JoinReturns
INNEROnly rows matching in both tables.
LEFT (OUTER)All left rows + matches; NULLs where no match.
RIGHT (OUTER)All right rows + matches.
FULL (OUTER)All rows from both, matched where possible.
CROSSCartesian product (every combination).
SELFA table joined to itself (e.g. employee → manager).
sql
SELECT c.name, o.total
FROM customers c
LEFT JOIN orders o ON o.customer_id = c.id;   -- keeps customers with no orders

09Grouping & Aggregates

sql
SELECT customer_id, COUNT(*) AS orders, SUM(total) AS spend
FROM orders
WHERE status = 'PAID'          -- filters rows (before grouping)
GROUP BY customer_id
HAVING SUM(total) > 1000;     -- filters groups (after aggregation)
WHERE vs HAVING

WHERE filters individual rows before grouping; HAVING filters aggregated groups after. Aggregate functions: COUNT SUM AVG MIN MAX.

10Subqueries & CTEs

sql
-- CTE: readable, reusable named subquery
WITH big_spenders AS (
  SELECT customer_id FROM orders GROUP BY customer_id HAVING SUM(total) > 1000
)
SELECT * FROM customers WHERE id IN (SELECT customer_id FROM big_spenders);
Correlated subquery

A subquery that references the outer query runs once per outer row — powerful but can be slow; a join or CTE is often faster.

PART 4

Indexes & Performance

11Indexes

An index is a separate data structure (usually a B-tree) that speeds up lookups and range scans — like a book's index — at the cost of slower writes and extra storage.

  • Clustered index — defines the physical row order; one per table (often the PK).
  • Non-clustered index — a separate structure pointing to rows; many allowed.
  • Composite index — on multiple columns; usable left-to-right (the "leftmost prefix" rule).
  • Covering index — contains all columns a query needs, so the table isn't touched.
When NOT to index

Indexes slow down INSERT/UPDATE/DELETE and cost storage. Avoid over-indexing, low-cardinality columns (few distinct values) for B-trees, and small tables. Index columns used in WHERE, JOIN, and ORDER BY.

12Reading a Query Plan

Use EXPLAIN (or EXPLAIN ANALYZE) to see how the optimizer will run a query.

  • Full/sequential scan — reads every row; fine for small tables, a red flag on big ones with selective filters.
  • Index scan / seek — uses an index; usually what you want for selective queries.
  • Watch row estimates vs actuals (stale statistics mislead the optimizer) and expensive joins/sorts.
PART 5

Transactions & Concurrency

13ACID

PropertyGuarantee
AtomicityAll statements succeed or none do (all-or-nothing).
ConsistencyA transaction moves the DB from one valid state to another (constraints hold).
IsolationConcurrent transactions don't corrupt each other.
DurabilityOnce committed, data survives crashes (written to disk/log).

14Isolation Levels & Read Anomalies

LevelDirty readNon-repeatablePhantom
READ UNCOMMITTEDPossiblePossiblePossible
READ COMMITTEDNoPossiblePossible
REPEATABLE READNoNoPossible*
SERIALIZABLENoNoNo
  • Dirty read — reading another transaction's uncommitted change.
  • Non-repeatable read — a row's value changes between two reads.
  • Phantom read — new rows appear matching a repeated query. (*InnoDB's REPEATABLE READ blocks phantoms via gap locks.)
Tradeoff

Higher isolation = fewer anomalies but more locking/less concurrency. Most systems default to READ COMMITTED (Oracle, Postgres) or REPEATABLE READ (MySQL InnoDB).

15Locks & Deadlocks

Databases use shared (read) and exclusive (write) locks at row or table granularity. A deadlock is two transactions each waiting on a lock the other holds.

Avoiding deadlocks

Access tables/rows in a consistent order, keep transactions short, and use appropriate isolation. The DB detects deadlocks and kills one transaction as the "victim" — your app should catch and retry it.

PART 6

Advanced SQL

16Views, Stored Procedures & Triggers

  • View — a named, saved query; a virtual table. Simplifies access and hides complexity. A materialized view stores the result for speed.
  • Stored procedure — precompiled routine of SQL/procedural code run on the server; reduces round-trips.
  • Trigger — code that runs automatically on INSERT/UPDATE/DELETE; used for auditing or enforcing rules. Use sparingly — hidden logic is hard to debug.

17Window Functions

Window functions compute across a set of rows related to the current row without collapsing them like GROUP BY does.

sql
SELECT name, dept, salary,
       RANK() OVER (PARTITION BY dept ORDER BY salary DESC) AS rank_in_dept,
       AVG(salary) OVER (PARTITION BY dept) AS dept_avg
FROM employees;
Common ones

ROW_NUMBER, RANK, DENSE_RANK, LAG, LEAD, SUM/AVG OVER — great for running totals, rankings, and comparing a row to its neighbours.

PART 7

Quick-Fire Q&A

Tap a question to reveal the answer.

>DBMS vs RDBMS?

A DBMS manages data; an RDBMS stores it as related tables with constraints and SQL. All RDBMS are DBMS, not vice versa.

>Primary key vs unique key?

A primary key is NOT NULL, unique, and one per table; a unique key enforces uniqueness but typically allows one NULL and you can have several.

>DELETE vs TRUNCATE vs DROP?

DELETE removes rows (filterable, rollbackable); TRUNCATE fast-removes all rows (DDL, usually not rollbackable); DROP removes the table entirely.

>What is normalization and why?

Structuring tables to eliminate redundancy and anomalies. 1NF atomic, 2NF no partial dependency, 3NF no transitive dependency.

>WHERE vs HAVING?

WHERE filters rows before grouping; HAVING filters groups after aggregation.

>INNER vs LEFT JOIN?

INNER returns only matching rows; LEFT returns all left rows with NULLs where the right has no match.

>What does an index cost?

Faster reads for selective queries, but slower writes and extra storage. Index selective columns used in filters/joins/sorts.

>Explain ACID.

Atomicity (all-or-nothing), Consistency (valid states), Isolation (no interference), Durability (survives crashes).

>What is a dirty read?

Reading data another transaction has modified but not yet committed; prevented at READ COMMITTED and above.

>How do you handle a deadlock?

The DB kills one transaction as the victim; the app catches the error and retries. Prevent by consistent lock ordering and short transactions.

>GROUP BY vs window function?

GROUP BY collapses rows into one per group; a window function computes across related rows while keeping each row.

Final tips for the interview

  1. Nail normalization and keys — the most common DB opener.
  2. Explain the logical order of execution to reason about queries.

  5. Explain ACID and isolation levels with the three read anomalies.
  6. Mention deadlock handling (retry) and window functions for analytics.
