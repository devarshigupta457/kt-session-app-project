PART 1

Foundations & Auto-Configuration

01Spring Boot vs Spring

Spring Framework gives you the core container, DI, AOP and MVC — but wiring it up historically meant heavy XML/Java config. Spring Boot sits on top and removes that friction with three ideas:

  • Opinionated auto-configuration — sensible defaults based on what is on the classpath.
  • Starter dependencies — one coordinate pulls a curated, version-aligned set.
  • Embedded servers — ship a runnable fat JAR; no external Tomcat to deploy.
One-liner

"Spring Boot is Spring with convention-over-configuration, auto-config, starters and an embedded server — so you write business code, not boilerplate."

02Starters & the Build File

A starter is a dependency descriptor bundling everything you typically need for a capability, with versions managed by the Boot BOM (so you rarely specify versions yourself).

xml
<parent>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-parent</artifactId>
  <version>3.3.0</version>   <!-- manages all versions -->
</parent>

<dependencies>
  <dependency><artifactId>spring-boot-starter-web</artifactId></dependency>
  <dependency><artifactId>spring-boot-starter-data-jpa</artifactId></dependency>
</dependencies>

Common starters: -web (Spring MVC + Tomcat), -data-jpa, -security, -validation, -actuator, -test.

03@SpringBootApplication

The single annotation on your main class is a meta-annotation combining three:

java
// @SpringBootApplication  ==
@Configuration          // this class can declare @Bean methods
@EnableAutoConfiguration // turn on Boot's auto-config machinery
@ComponentScan         // scan this package + sub-packages for beans
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}
Gotcha

Component scanning starts from the main class's package. Put it at the root package so all sub-packages are scanned; beans in a sibling or parent package won't be found.

04How Auto-Configuration Actually Works

This is a favourite deep-dive question. @EnableAutoConfiguration asks Boot to load candidate configuration classes and apply the ones whose conditions match.

  1. Boot reads the list of auto-configuration classes from each jar's META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports (pre-2.7 this was spring.factories).
  2. Each class is guarded by @Conditional annotations, so it only applies when appropriate: @ConditionalOnClass, @ConditionalOnMissingBean, @ConditionalOnProperty.
  3. Example: if DataSource is on the classpath and you haven't defined one, DataSourceAutoConfiguration creates one from your properties.
java
@ConditionalOnClass(DataSource.class)
@ConditionalOnMissingBean(DataSource.class) // back off if user defined one
public DataSource dataSource(DataSourceProperties props) { ... }
Debug it

Run with --debug to print the condition evaluation report showing which auto-configs matched and why. Exclude one with @SpringBootApplication(exclude = XxxAutoConfiguration.class).

05SpringApplication & the Startup Sequence

SpringApplication.run() creates the ApplicationContext, and in rough order: prepares the environment (properties, profiles), fires listeners/events, creates the context, registers bean definitions, runs auto-config, instantiates singleton beans, starts the embedded server, then invokes any CommandLineRunner/ApplicationRunner beans.

Run code at startup

Implement ApplicationRunner or CommandLineRunner to execute logic once the context is ready — useful for seeding data or launching a batch job.

PART 2

IoC Container & Dependency Injection

06Inversion of Control & Beans

IoC means the framework, not your code, creates and wires objects. The ApplicationContext (the IoC container) manages beans — objects it instantiates, configures and injects.

DI is how IoC is achieved: dependencies are handed to a class rather than the class constructing them. Benefits: loose coupling, easy testing (swap in mocks), single responsibility.

07Stereotypes & @Bean

AnnotationMeaning
@ComponentGeneric Spring-managed bean.
@ServiceBusiness-logic layer (a @Component semantically).
@RepositoryPersistence layer; also translates DB exceptions.
@Controller / @RestControllerWeb layer.
@Configuration + @BeanProgrammatic bean definitions (for third-party classes you cannot annotate).
java
@Configuration
public class AppConfig {
    @Bean
    public RestClient restClient() {   // external type -> declare with @Bean
        return RestClient.create();
    }
}

08Injection Styles & Resolving Ambiguity

java
@Service
public class OrderService {
    private final OrderRepository repo;
    // CONSTRUCTOR injection (preferred) -- no @Autowired needed on a single ctor
    public OrderService(OrderRepository repo) { this.repo = repo; }
}
Why constructor injection

It allows final fields (immutability), makes dependencies explicit, fails fast if a bean is missing, and needs no reflection to test. Field injection (@Autowired on a field) hides dependencies and can't be made final.

When multiple beans of a type exist: mark one @Primary, or select by name with @Qualifier("beanName").

09Bean Scopes & Lifecycle

ScopeInstances
singleton (default)One per container.
prototypeA new one on every injection/lookup.
request / sessionOne per HTTP request / session (web).

Lifecycle callbacks: @PostConstruct runs after injection; @PreDestroy before shutdown. Use @Lazy to defer creation until first use.

Classic trap

Injecting a prototype bean into a singleton gives you the same instance forever (injected once). Use a provider (ObjectProvider) or method injection to get a fresh one each time.

PART 3

Configuration & Profiles

10Externalized Configuration

Boot reads configuration from many sources with a defined precedence (later wins): defaults < application.yml < profile-specific files < environment variables < command-line args. This lets the same JAR run in every environment.

java
@Value("${app.page-size:20}")   // inject single value with default
private int pageSize;

11@ConfigurationProperties (type-safe config)

For grouped settings, bind a whole block to a typed object instead of scattering @Value.

java
@ConfigurationProperties(prefix = "app.mail")
public record MailProps(String host, int port, boolean tls) {}
// app.mail.host / app.mail.port / app.mail.tls  ->  bound automatically
@Value vs @ConfigurationProperties

Use @Value for one-off values; use @ConfigurationProperties for a group — it is type-safe, supports validation (@Validated), relaxed binding (page-size = pageSize), and nested objects.

12Profiles

Profiles let you swap configuration and beans per environment.

yaml
# application.yml
spring:
  profiles:
    active: dev
# application-dev.yml, application-prod.yml hold overrides

Annotate beans with @Profile("prod") to include them only in that profile. Activate via spring.profiles.active, an env var, or --spring.profiles.active=prod.

PART 4

Web Layer & REST

13REST Controllers

java
@RestController              // = @Controller + @ResponseBody
@RequestMapping("/api/orders")
public class OrderController {
    private final OrderService service;
    public OrderController(OrderService s) { this.service = s; }

    @GetMapping("/{id}")
    public ResponseEntity<OrderDto> get(@PathVariable Long id) {
        return ResponseEntity.ok(service.find(id));
    }

    @PostMapping
    public ResponseEntity<OrderDto> create(@Valid @RequestBody CreateOrder body) {
        OrderDto saved = service.create(body);
        return ResponseEntity.status(HttpStatus.CREATED).body(saved);
    }
}
Why ResponseEntity

It gives you explicit control over the HTTP status and headers, not just the body. Return 201 Created on POST, 204 No Content on delete, 404 when not found.

14Request Mapping Cheatsheet

AnnotationBinds
@PathVariableA URI segment: /orders/{id}
@RequestParamA query param: ?page=2
@RequestBodyThe JSON body → an object (via Jackson)
@RequestHeaderA header value
@GetMapping etc.Shortcuts for @RequestMapping(method=...)

15Global Exception Handling

Centralise error handling with @RestControllerAdvice so controllers stay clean and responses are consistent.

java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(NotFoundException.class)
    @ResponseStatus(HttpStatus.NOT_FOUND)
    public ApiError handleNotFound(NotFoundException e) {
        return new ApiError("NOT_FOUND", e.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public ApiError handleValidation(MethodArgumentNotValidException e) {
        return new ApiError("VALIDATION", e.getMessage());
    }
}

16Bean Validation

java
public record CreateOrder(
    @NotBlank String customerId,
    @Positive BigDecimal amount,
    @Email String email) {}

Add @Valid on the @RequestBody to trigger validation; failures throw MethodArgumentNotValidException, which your advice maps to 400. Common annotations: @NotNull, @NotBlank, @Size, @Min/@Max, @Pattern.

PART 5

Data Layer — Spring Data JPA

17Entities & Repositories

java
@Entity
@Table(name = "ORDERS")
public class Order {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    private String customerId;
    private BigDecimal amount;
}

public interface OrderRepository extends JpaRepository<Order, Long> {
    List<Order> findByCustomerId(String customerId);      // derived query
    List<Order> findByAmountGreaterThanOrderByAmountDesc(BigDecimal min);
}

JpaRepository gives you save, findById, findAll, delete, paging and sorting for free. Derived queries are generated from method names (findBy, And, OrderBy, Between…).

18Custom Queries, Paging & Projections

java
@Query("SELECT o FROM Order o WHERE o.amount > :min")          // JPQL
List<Order> bigOrders(@Param("min") BigDecimal min);

@Query(value = "SELECT * FROM ORDERS WHERE ROWNUM <= 10", nativeQuery = true)
List<Order> topTen();                                        // native SQL (Oracle)

Page<Order> findByCustomerId(String id, Pageable pageable);  // paging + sorting
DTO projection

Return an interface or record projection instead of the full entity to fetch only the columns you need — less data, no lazy-loading surprises.

19Transactions

@Transactional wraps a method in a database transaction: commit on success, rollback on a runtime exception.

java
@Transactional
public void transfer(Long from, Long to, BigDecimal amt) {
    debit(from, amt);
    credit(to, amt);   // if this throws, the debit rolls back too
}
  • Propagation (REQUIRED default, REQUIRES_NEW, NESTED) — how a method joins/starts a transaction.
  • Isolation controls concurrent-read visibility (READ_COMMITTED, etc.).
  • By default it rolls back on unchecked exceptions only; use rollbackFor for checked ones.
Self-invocation trap

@Transactional works via a proxy, so calling one @Transactional method from another method in the same class bypasses the proxy and the annotation is ignored.

20The N+1 Problem & Fetch Strategies

A frequent senior question. With lazy associations, loading N parents and then touching each one's children fires 1 + N queries.

java
// BAD: 1 query for orders + N queries for each order.items
orders.forEach(o -> o.getItems().size());

// FIX: fetch join in one query
@Query("SELECT o FROM Order o JOIN FETCH o.items WHERE o.id IN :ids")
List<Order> withItems(@Param("ids") List<Long> ids);

Fixes: JOIN FETCH, an @EntityGraph, or batch fetching (hibernate.default_batch_fetch_size). Prefer LAZY by default and fetch deliberately.

21HikariCP & Oracle Notes

Boot uses HikariCP as the default connection pool. Tune it via properties:

yaml
spring:
  datasource:
    url: jdbc:oracle:thin:@//host:1521/servicename
    username: app
    password: "${DB_PASSWORD}"
    hikari:
      maximum-pool-size: 10
      minimum-idle: 2
      connection-timeout: 30000
      schema: APP_SCHEMA          # default schema for the session
  jpa:
    hibernate.ddl-auto: validate    # never 'update' in production
    properties.hibernate.dialect: org.hibernate.dialect.OracleDialect
Pool sizing

Bigger is not better — a pool larger than the DB can serve just queues work. Start small (about the number of DB cores) and measure. Set ddl-auto=validate so Hibernate checks the schema but never mutates it.

PART 6

Cross-Cutting, Observability & Security

22Aspect-Oriented Programming

AOP extracts cross-cutting concerns (logging, metrics, security, transactions) out of business code into aspects applied via proxies.

java
@Aspect @Component
public class TimingAspect {
    @Around("@annotation(Timed)")
    public Object time(ProceedingJoinPoint pjp) throws Throwable {
        long t = System.nanoTime();
        try { return pjp.proceed(); }
        finally { log.info("{} took {}ns", pjp.getSignature(), System.nanoTime()-t); }
    }
}

Advice types: @Before, @After, @AfterReturning, @AfterThrowing, @Around (most powerful). @Transactional itself is AOP.

23Actuator & Metrics

spring-boot-starter-actuator exposes operational endpoints for health, metrics and info.

yaml
management:
  endpoints.web.exposure.include: health,info,metrics,prometheus
  endpoint.health.show-details: when-authorized
  • /actuator/health — liveness/readiness probes (great for Kubernetes).
  • /actuator/metrics — JVM, HTTP, datasource metrics via Micrometer, which can export to Prometheus/Grafana.
Secure it

Actuator endpoints can leak internals — restrict exposure and protect them with Spring Security in production.

@WebMvcTest(OrderController.class)
class OrderControllerTest {
    @Autowired MockMvc mvc;

    @Test void returns404() throws Exception {
        given(service.find(9L)).willThrow(new NotFoundException());
        mvc.perform(get("/api/orders/9")).andExpect(status().isNotFound());
    }
}
Real DB tests

Use Testcontainers to spin up a real Oracle/Postgres in Docker for integration tests — far more faithful than an in-memory H2.

25Spring Security & JWT (overview)

Spring Security is a chain of servlet filters. Two core ideas: authentication (who are you) and authorization (what may you do).

java
@Bean
public SecurityFilterChain chain(HttpSecurity http) throws Exception {
    return http
        .csrf(csrf -> csrf.disable())               // stateless API
        .authorizeHttpRequests(a -> a
            .requestMatchers("/api/public/**").permitAll()
            .anyRequest().authenticated())
        .oauth2ResourceServer(o -> o.jwt(Customizer.withDefaults()))
        .build();
}
Stateless JWT flow

Client logs in → server returns a signed JWT → client sends it in Authorization: Bearer <token> → a filter validates the signature/expiry and sets the security context. No server-side session, so it scales horizontally. Use method security (@PreAuthorize("hasRole('ADMIN')")) for fine-grained rules.

PART 7

Quick-Fire Q&A

Tap a question to reveal the answer.

>What does @SpringBootApplication do?

It combines @Configuration, @EnableAutoConfiguration, and @ComponentScan — enabling bean definitions, auto-config, and scanning from the main package.

>How does auto-configuration work?

Boot loads candidate config classes listed in each jar's AutoConfiguration.imports and applies each one only if its @Conditional guards match (class present, no user bean, property set).

>Constructor vs field injection?

Constructor injection is preferred: it allows final fields, makes dependencies explicit and testable, and fails fast. Field injection hides dependencies and needs reflection to test.

>Singleton vs prototype scope?

Singleton (default) is one instance per container; prototype creates a new one per request. Injecting a prototype into a singleton captures a single instance unless you use a provider.

>How do you handle exceptions globally?

With @RestControllerAdvice + @ExceptionHandler methods mapping exceptions to HTTP statuses and a consistent error body.

>What is the N+1 problem and how do you fix it?

Lazy associations cause 1 query for parents plus N for each child. Fix with JOIN FETCH, an @EntityGraph, or batch fetching.

>When does @Transactional NOT roll back?

By default only on unchecked exceptions. Also, self-invocation (calling the method from within the same class) bypasses the proxy, so the annotation is ignored.

>@Value vs @ConfigurationProperties?

@Value injects a single property; @ConfigurationProperties binds a group to a typed object with validation, relaxed binding, and nesting.

>How do profiles work?

Profile-specific property files and @Profile beans activate based on spring.profiles.active, letting one artifact behave differently per environment.

>What is Actuator for?

Production-ready endpoints for health, metrics (via Micrometer), and info — used for monitoring and Kubernetes probes.

>How does stateless JWT auth work in Spring Security?

A filter validates the Bearer token's signature and expiry on each request and populates the security context — no server session, so it scales horizontally.

>Why prefer DTOs over returning entities?

DTOs decouple the API from the schema, avoid lazy-loading serialization issues, and expose only the needed fields.

Final tips for the interview

  1. Explain auto-configuration and @Conditional — the signature Boot question.
  2. Default to constructor injection and say why.
  3. Know @Transactional semantics, propagation, and the self-invocation trap.
  4. Be ready for the N+1 problem and fetch strategies.
  5. Talk through a clean layered design: controller → service → repository → DTO.
  6. Mention Actuator, Micrometer, and Testcontainers — signals production maturity.
