PART 1

The Approach

01What Is System Design?

System design is planning how the pieces of a large application fit together so it stays fast, reliable, and able to grow — before writing code. In interviews you're asked to design something like "a URL shortener" or "Instagram" and reason about the tradeoffs.

Layman analogy

It's like designing a city, not a single house. You plan roads (network), water and power (infrastructure), traffic flow (load), and room to add neighbourhoods later (scaling). There's rarely one "right" design — only tradeoffs.

02A Framework for the Interview

Don't jump to boxes and arrows. Follow a structure so you look methodical:

1 Clarify requirements what must it do? how big? 2 Estimate scale users, reads/writes per second, storage 3 Define the API the main operations 4 High-level design draw the main components 5 Deep dive data model, one hard part in detail 6 Find bottlenecks scale, cache, replicate, shard; discuss tradeoffs
Key habit

Talk out loud and state assumptions. Interviewers score your reasoning and tradeoffs, not a memorised "correct" diagram.

03Requirements & Back-of-Envelope Estimation

  • Functional requirements — what it does ("shorten a URL, redirect to the original").
  • Non-functional requirements — how well: scale, latency, availability, consistency.

Then do rough maths to size the system. Example for a URL shortener:

estimation
100M new URLs/day  ->  100M / 86,400s  ~= 1,160 writes/sec
Reads are ~100x writes (redirects)  ->  ~116,000 reads/sec
Store 100M/day * 365 * 5 years ~= 180B records
Each record ~500 bytes  ->  ~90 TB over 5 years
Why estimate?

The numbers decide the design. 100 reads/sec fits one box; 100,000 reads/sec needs caching, replicas, and load balancing. Read-heavy vs write-heavy changes everything.

PART 2

Scaling

04Vertical vs Horizontal Scaling

Vertical (scale up)Horizontal (scale out)
HowBigger machine (more CPU/RAM)More machines
LimitHardware ceiling; single point of failureNear-unlimited; needs coordination
CostSimple but pricey at the topCheap commodity boxes
Layman analogy

Vertical = replacing your delivery van with a bigger truck. Horizontal = buying more vans. Big systems scale out because you can always add another van, but you can't buy an infinitely large truck — and if the one truck breaks, you're stuck.

05Load Balancing

A load balancer sits in front of many identical servers and spreads incoming requests so no single server is overwhelmed — and routes around any that fail.

Layman analogy

The queue manager at a bank or supermarket: "next customer to till 3, please." Nobody waits behind one slow cashier while others sit idle.

  • Algorithms: round-robin, least-connections, IP-hash.
  • Layer 4 (transport) is fast and simple; Layer 7 (application) can route by URL/headers (send /images to one pool, /api to another).
  • Run the load balancer itself redundantly so it isn't a single point of failure.

06Stateless Services

For horizontal scaling to work, app servers should be stateless — they keep no per-user data in local memory, so any server can handle any request.

Why it matters

If a user's session lived only on server A, the load balancer would have to keep sending them to A ("sticky sessions") — and if A dies, they're logged out. Instead, store session/state in a shared place (a cache like Redis or the DB). Then servers are interchangeable, like identical cashiers — it doesn't matter which one serves you.

PART 3

Making It Fast

07Caching

A cache keeps frequently used data in fast memory so you avoid hitting the slow database every time. It's often the single biggest performance win.

Layman analogy

Keeping your most-used files on your desk instead of walking to the archive room each time. The desk is small but instant; the archive is huge but slow.

StrategyHow it works
Cache-aside (lazy)App checks cache; on a miss, reads the DB and fills the cache. Most common.
Write-throughWrites go to cache and DB together — cache always fresh, writes slower.
Write-backWrite to cache now, DB later — fast, but risk of loss on crash.
The two hard parts

Eviction: when full, drop items — usually LRU (Least Recently Used). Invalidation: keeping the cache from serving stale data. "There are only two hard things in computer science: cache invalidation and naming things." Tools: Redis, Memcached.

08CDN (Content Delivery Network)

A CDN caches static content (images, video, CSS/JS) on servers spread around the world, so users download from a location near them instead of your one central server.

Layman analogy

Instead of one giant warehouse everyone drives to, a chain of local convenience stores. You grab milk from the shop down the street — much faster than a cross-country trip — and it takes load off the central warehouse.

PART 4

Storing Data

09SQL vs NoSQL

SQL (relational)NoSQL
SchemaFixed, structuredFlexible
StrengthsJoins, transactions, consistencyScale-out, high write throughput, flexible data
TypesPostgres, MySQL, OracleDocument, key-value, column, graph
Use whenRelationships & strong consistency matter (payments)Huge scale, simple lookups, evolving schema (feeds, logs)
Answer well

Don't say "NoSQL is better." Say: "Use SQL when you need transactions and relationships; use NoSQL when you need massive scale or a flexible schema for a specific access pattern." Many real systems use both (polyglot persistence).

10Replication (Copies for Safety & Reads)

Replication keeps copies of the data on multiple servers. A common setup is leader-follower: writes go to the leader, which copies changes to followers; reads can be served by followers.

Layman analogy

Photocopies of an important document kept in several offices. If the original is lost, a copy survives (durability), and many people can read the copies at once (scaling reads).

Replication lag

Copying isn't instant, so a follower may be slightly behind the leader. Reading your own just-made change from a follower might show old data — route critical reads to the leader, or accept eventual consistency.

11Sharding (Partitioning)

When one database can't hold or serve all the data, shard it: split rows across multiple databases by a shard key (e.g. user ID). Each shard holds a slice.

Layman analogy

Splitting one huge phone book into volumes: A–M in one book, N–Z in another. To find "Smith," you go straight to the second book — smaller and faster to search.

The tradeoffs

Pick the shard key carefully: a bad key creates hotspots (one shard gets all the traffic). Queries spanning shards (and joins) get harder, and rebalancing when you add shards is painful. Consistent hashing helps reduce reshuffling.

PART 5

The Big Tradeoff

12The CAP Theorem

CAP says a distributed data store can't perfectly guarantee all three at once:

  • Consistency — every read sees the latest write.
  • Availability — every request gets a (non-error) response.
  • Partition tolerance — keeps working when the network between nodes breaks.
The real meaning

Networks will partition, so P is non-negotiable. The real choice during a partition is C or A: refuse to answer to stay consistent (CP, e.g. a bank), or answer with possibly-stale data to stay available (AP, e.g. a social feed).

Layman analogy

Two shop tills lose their link to each other. You either stop selling the last item so you never oversell (consistency), or keep selling and risk two customers buying the same last item (availability). You can't have both while the link is down.

13Consistency Models

  • Strong consistency — after a write, everyone immediately sees the new value. Needed for money/inventory. Costs latency and availability.
  • Eventual consistency — copies converge "soon"; reads may briefly be stale. Fine for likes, view counts, feeds. Enables high availability and scale.
Pick per feature

A single product can mix both: strong consistency for the checkout/payment, eventual consistency for the "1.2k likes" counter. Match the model to the cost of being wrong.

PART 6

Decoupling & Reliability

14Message Queues (Asynchronous Processing)

A message queue (or log like Kafka) lets a service hand off work and move on, while another service processes it at its own pace. This decouples producers from consumers and smooths out traffic spikes.

Layman analogy

The order tickets on a rail in a restaurant kitchen. Waiters clip up orders and keep serving; the kitchen works through them at its speed. A rush doesn't overwhelm the cooks — tickets just queue up (this is back-pressure handling).

Uses: sending emails, generating reports, processing uploads, evening out spikes, and connecting microservices via events. Trade instant results for scalability and resilience.

15Reliability & Availability

  • Redundancy — no single point of failure; run at least two of everything, ideally across data centres/zones.
  • Failover — if the leader dies, promote a replica automatically.
  • Availability "nines": how much downtime you allow per year.
AvailabilityDowntime / year
99% (two nines)~3.65 days
99.9% (three nines)~8.76 hours
99.99% (four nines)~52 minutes
99.999% (five nines)~5 minutes
Cost curve

Each extra nine is much more expensive. Match the target to the business — a bank needs more nines than a blog.

PART 7

Numbers & a Worked Example

16Numbers Every Engineer Should Know

Rough latencies to reason about where time goes (orders of magnitude, not exact):

read from memory (RAM) ~100 nanoseconds read from SSD ~100 microseconds (1,000x slower) read from disk (HDD) ~10 milliseconds (100,000x slower) network round trip same region ~0.5 ms network round trip cross-continent ~100-150 ms
Takeaway

Memory is thousands of times faster than disk, and a cross-region network hop dwarfs local work. This is why we cache in memory, keep data close to users (CDN), and avoid chatty cross-region calls.

17Worked Example: A URL Shortener

Tying it together in plain terms — design a "TinyURL":

  1. API: POST /shorten (long URL → short code) and GET /{code} (redirect to the long URL).
  2. Scale (from earlier): read-heavy — ~1k writes/sec, ~116k reads/sec.
  3. Generate the code: encode a unique counter into base62 (a–z, A–Z, 0–9) to get a short 6–7 character key.
  4. Store: a simple key→value mapping (code → long URL) — NoSQL key-value store fits perfectly and shards well by code.
  5. Make reads fast: put a cache (Redis) in front — hot links are served from memory; the DB is only hit on a cache miss.
  6. Scale out: a load balancer in front of many stateless app servers; replicate and shard the store as it grows.
  7. Reliability: redundant servers and DB replicas so no single failure takes it down.
Notice the pattern

Every earlier concept showed up: estimate → API → data model → cache → load balance → replicate/shard → make it reliable. That's the template for almost any design question.

PART 8

Quick-Fire Q&A

Tap a question to reveal the answer.

>How do you approach a system design question?

Clarify requirements, estimate scale, define the API, sketch a high-level design, deep-dive one hard part, then find and fix bottlenecks — talking through tradeoffs.

>Vertical vs horizontal scaling?

Vertical = a bigger machine (simple, has a ceiling, single point of failure); horizontal = more machines (scales further, needs load balancing and stateless services).

>Why must app servers be stateless to scale?

So any server can handle any request. State (sessions) goes in a shared store like Redis, so servers are interchangeable and one dying doesn't lose user data.

>What does a load balancer do?

Distributes requests across identical servers and routes around failed ones — like a queue manager sending you to a free till.

>Why is caching so effective?

Memory is thousands of times faster than disk, and cached items skip the database entirely. It's usually the biggest quick win for read-heavy systems.

>What are the hard problems with caches?

Eviction (dropping items when full, usually LRU) and invalidation (avoiding stale data).

>SQL or NoSQL — how do you choose?

SQL for relationships, transactions, and strong consistency; NoSQL for massive scale, high write throughput, or flexible schema. Often both in one system.

>Explain sharding and its risks.

Splitting data across databases by a shard key. Risks: hotspots from a bad key, harder cross-shard queries, and painful rebalancing when adding shards.

>What is the CAP theorem really saying?

Since network partitions happen, you must choose during one: stay consistent but possibly unavailable (CP), or stay available but possibly serve stale data (AP).

>Strong vs eventual consistency?

Strong: everyone sees the latest write immediately (money). Eventual: copies converge soon, brief staleness allowed (likes, feeds).

>Why use a message queue?

To decouple services and absorb spikes — the producer hands off work and moves on while a consumer processes at its own pace.

>What does 99.99% availability mean?

About 52 minutes of downtime per year. Each additional nine costs significantly more, so match it to business need.

Final tips for the interview

  1. Always clarify requirements and estimate scale first — the numbers drive design.
  2. Default to horizontal scaling + stateless services + a load balancer.
  3. Reach for caching and CDNs early on read-heavy systems.
  4. Justify SQL vs NoSQL and explain replication vs sharding.
  5. Use CAP to frame the consistency/availability tradeoff per feature.
  6. Talk tradeoffs out loud — that, not a perfect diagram, is what's scored.
