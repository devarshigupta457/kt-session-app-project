PART 1

Foundations

01What Spring Batch Is & When to Use It

Spring Batch is a framework for robust bulk processing — reading large volumes of records, transforming them, and writing them out, with first-class support for transactions, restart, skip/retry, logging and scaling.

Typical jobs: nightly ETL, data migrations, generating reports/statements, importing files into a database, reconciling systems, purging/archiving old data.

Why not a plain loop?

A hand-rolled loop over a million rows gives you no restart-after-failure, no commit batching, no skip/retry policy, no progress metadata, and no scaling model. Spring Batch provides all of that as infrastructure.

02Batch vs Streaming (and Kafka)

Spring BatchStreaming (e.g. Kafka)
DataBounded, finite datasetUnbounded, continuous events
TriggerScheduled / on-demand runAlways-on consumers
LatencyHigh (minutes/hours)Low (near real-time)
FitNightly ETL, migrations, reportsLive pipelines, event-driven
They combine

A common pattern: a batch job produces or consumes files/DB rows, while Kafka handles the real-time path. Choose batch when work is periodic and bounded.

PART 2

Core Architecture

03The Domain Model

Learn these relationships — they come up in almost every Spring Batch interview.

Job (a batch process definition) | run with JobParameters v JobInstance (a logical run, identified by job name + parameters) | v JobExecution (one attempt; a restart is a new execution of the same instance) | contains one or more v Step -> StepExecution (per-step attempt + read/write/skip counts)
  • Job = the whole process; composed of one or more Steps.
  • JobInstance = a specific logical run defined by job name + identifying JobParameters (e.g. date=2026-07-05). Re-running the same parameters resumes/blocks; new parameters = new instance.
  • JobExecution / StepExecution = a single attempt, holding status, timestamps, and counts.

04JobRepository, JobLauncher & Metadata Tables

  • JobLauncher starts a Job with JobParameters.
  • JobRepository persists all execution metadata (status, counts, context) so jobs are observable and restartable.

Spring Batch creates a set of metadata tables (BATCH_JOB_INSTANCE, BATCH_JOB_EXECUTION, BATCH_STEP_EXECUTION, BATCH_JOB_EXECUTION_CONTEXT…). Boot can auto-create them, but in production you usually run the vendor DDL yourself.

yaml
spring:
  batch:
    jdbc:
      initialize-schema: always   # dev; use 'never' + manual DDL in prod
    job:
      enabled: false             # don't auto-run jobs on startup

05ExecutionContext (state for restart)

Each Step and Job has an ExecutionContext — a persisted key/value store the framework uses to remember progress (e.g. how many rows were read) so a restarted job can pick up where it left off. You can also stash your own state in it.

Step vs Job context

Step-scoped context is for one step's progress; to pass data between steps, promote a key to the Job ExecutionContext (via an ExecutionContextPromotionListener).

PART 3

Processing Models

06Chunk-Oriented Processing

The core model: read items one by one, process each, and write them in chunks. Each chunk is a single transaction — commit every N items.

repeat until reader returns null: read item -> process item -> add to chunk buffer when buffer size == chunk(N): write the whole chunk -> COMMIT transaction update read/write counts in StepExecution
java
Step step = new StepBuilder("importStep", jobRepository)
    .<InputRow, OutputRow>chunk(500, transactionManager)  // commit interval
    .reader(reader())
    .processor(processor())
    .writer(writer())
    .build();
Chunk size tradeoff

Bigger chunks = fewer commits and higher throughput, but more memory and a larger rollback on failure. Tune it (often 100–1000) and measure. The processor may return null to filter an item out.

07Tasklet Steps

When a step is a single operation rather than an item pipeline (run a stored proc, delete a file, call a service, truncate a table), use a Tasklet.

java
Step cleanup = new StepBuilder("cleanup", jobRepository)
    .tasklet((contribution, chunkContext) -> {
        jdbc.update("DELETE FROM staging WHERE processed = 1");
        return RepeatStatus.FINISHED;
    }, transactionManager)
    .build();
Chunk or tasklet?

Chunk = process a collection of items with read/process/write. Tasklet = a single, self-contained unit of work.

08Readers, Processors & Writers

Rich set of built-in components; you rarely write I/O from scratch.

RoleBuilt-ins
ItemReaderFlatFileItemReader (CSV/fixed), JdbcCursorItemReader, JdbcPagingItemReader, JpaPagingItemReader, StaxEventItemReader (XML)
ItemProcessorYour transform/validate/filter logic; CompositeItemProcessor chains several
ItemWriterJdbcBatchItemWriter, JpaItemWriter, FlatFileItemWriter
java
@Bean
FlatFileItemReader<InputRow> reader() {
    return new FlatFileItemReaderBuilder<InputRow>()
        .name("reader").resource(new ClassPathResource("orders.csv"))
        .delimited().names("id", "customer", "amount")
        .targetType(InputRow.class).build();
}
@Bean
ItemProcessor<InputRow, OutputRow> processor() {
    return in -> in.amount() <= 0 ? null            // null => filtered out
                              : new OutputRow(in.id(), in.amount());
}
Restartable readers

Paging/cursor readers track their position in the ExecutionContext so a restart resumes cleanly. For scaling, prefer JdbcPagingItemReader (thread-safe) over cursor-based readers.

PART 4

Configuration & Launching

09Wiring a Job (Spring Batch 5 / Boot 3)

Modern config uses builders and injected JobRepository + PlatformTransactionManager. @EnableBatchProcessing is no longer required with Spring Boot 3 (Boot auto-configures the infrastructure).

java
@Configuration
public class ImportJobConfig {

    @Bean
    Job importJob(JobRepository repo, Step importStep) {
        return new JobBuilder("importJob", repo)
            .incrementer(new RunIdIncrementer())   // unique params each run
            .start(importStep)
            .build();
    }

    @Bean
    Step importStep(JobRepository repo, PlatformTransactionManager tx,
                    ItemReader<InputRow> r, ItemProcessor<InputRow,OutputRow> p,
                    ItemWriter<OutputRow> w) {
        return new StepBuilder("importStep", repo)
            .<InputRow, OutputRow>chunk(500, tx)
            .reader(r).processor(p).writer(w)
            .build();
    }
}

10Launching & JobParameters

Run a job from a Boot app, on a schedule, or from the command line. Each run needs unique identifying parameters or Batch will refuse to re-run a completed instance.

java
@Component
public class JobRunner implements ApplicationRunner {
    private final JobLauncher launcher; private final Job importJob;
    // constructor omitted
    public void run(ApplicationArguments args) throws Exception {
        JobParameters params = new JobParametersBuilder()
            .addString("date", LocalDate.now().toString())
            .addLong("run.id", System.currentTimeMillis())
            .toJobParameters();
        launcher.run(importJob, params);
    }
}
Scheduling

Trigger with @Scheduled(cron="0 0 2 * * *") for nightly runs, or Quartz for clustered scheduling. A JobParametersIncrementer (like RunIdIncrementer) auto-varies parameters so the same job can run repeatedly.

PART 5

Reliability & Fault Tolerance

11Restartability

If a job fails midway, re-launching the same JobInstance resumes from the last committed chunk — completed steps are skipped and the failed step continues using its saved ExecutionContext.

Key rule

Restart works because each chunk commits its progress transactionally. Make a step non-restartable with .allowStartIfComplete(false), or cap attempts with .startLimit(n). Design writers to be idempotent so a resumed chunk can't double-write.

12Skip & Retry

Fault-tolerant steps let a job survive bad records or transient errors instead of aborting.

java
new StepBuilder("step", repo)
    .<In,Out>chunk(500, tx).reader(r).processor(p).writer(w)
    .faultTolerant()
    .skip(FlatFileParseException.class).skipLimit(50)   // tolerate up to 50 bad rows
    .retry(DeadlockLoserDataAccessException.class).retryLimit(3) // transient DB errors
    .build();
  • Skip — discard an item that throws a listed exception, up to a limit; log it via a skip listener.
  • Retry — re-attempt an item on transient failures (deadlocks, timeouts) before giving up.

13Listeners

Hook into the lifecycle for logging, metrics, notifications, and cleanup.

ListenerFires
JobExecutionListenerbefore/after the whole job
StepExecutionListenerbefore/after a step
ChunkListeneraround each chunk (incl. errors)
ItemReadListener / ItemProcessListener / ItemWriteListeneraround each item op
SkipListenerwhen an item is skipped — record the bad record
PART 6

Scaling & Performance

14Scaling Strategies

When one thread is too slow, Spring Batch offers several models, in rough order of complexity:

StrategyWhat it does
Multi-threaded stepProcess chunks on a thread pool within one step. Simple, but the reader must be thread-safe and ordering is lost.
Parallel stepsRun independent steps concurrently via a split flow.
Async processor/writerAsyncItemProcessor processes items on a pool; the writer unwraps futures.
PartitioningSplit data into partitions, each handled by its own step instance (local threads or remote workers).
Remote chunkingA master reads; workers process/write over messaging. Network-heavy.

15Partitioning

The most scalable model: a Partitioner divides the input (e.g. by ID range or file) into partitions, and a PartitionHandler runs a worker step for each — locally on threads or remotely across nodes.

Manager Step Partitioner -> creates {p0: ids 1-10k, p1: 10k-20k, ...} | distributes to v Worker Step x N (each a full reader/processor/writer over its slice)
Why it scales best

Each partition has an independent ExecutionContext, so partitions restart and scale independently. Ideal for large datasets that split cleanly by key range or file.

PART 7

Quick-Fire Q&A

Tap a question to reveal the answer.

>What problem does Spring Batch solve?

Robust bulk processing: transactional chunking, restart-after-failure, skip/retry, metadata/monitoring, and scaling — things a plain loop lacks.

>Job vs JobInstance vs JobExecution?

Job is the definition; JobInstance is a logical run identified by job name + identifying parameters; JobExecution is a single attempt of that instance (a restart is a new execution).

>What is chunk-oriented processing?

Read-process-write with items grouped into chunks; each chunk of N items is written and committed in one transaction.

>Chunk vs tasklet?

Chunk processes a stream of items (reader/processor/writer); tasklet is a single self-contained operation like running a query or deleting a file.

>How does restart work?

Each chunk commits progress to the JobRepository and ExecutionContext, so re-launching the same instance resumes from the last committed point and skips completed steps.

>Skip vs retry?

Skip discards bad items (bad data) up to a limit; retry re-attempts items on transient errors (deadlocks/timeouts) before failing.

>What does the JobRepository do?

Persists all job/step execution metadata and context into the BATCH_* tables, enabling monitoring and restart.

>How do you ensure a job can re-run?

Vary identifying JobParameters (e.g. a RunIdIncrementer or a date) — Batch blocks re-running a completed instance with identical parameters.

>What scaling options exist?

Multi-threaded step, parallel steps, async processor/writer, partitioning (local or remote), and remote chunking — partitioning scales best.

>Why make writers idempotent?

On restart a partially processed chunk may be re-run, so idempotent writes prevent duplicates.

Final tips for the interview

  1. Draw the Job/Instance/Execution + Step model — the anchor question.
  2. Explain the chunk transaction and how commit interval affects throughput vs memory.
  3. Distinguish chunk vs tasklet with concrete examples.
  4. Nail restartability: committed chunks + idempotent writers.
  5. Separate skip (bad data) from retry (transient errors).
  6. Reach for partitioning when asked how to scale a huge dataset.
