PART 1

Foundations

01Overview

PostgreSQL is a powerful, standards-compliant open-source RDBMS often called "the most advanced open-source database." It is prized for correctness, extensibility, rich data types, and strong SQL feature support.

Process model

Postgres forks a separate backend process per connection (managed by the postmaster), so a connection pooler (PgBouncer) is important at scale.

02Rich Data Types

PART 2

SQL & Features

03SQL Features

sql
-- RETURNING: get generated values back in the same statement
INSERT INTO orders(customer_id, total) VALUES (7, 99) RETURNING id;

-- UPSERT: ON CONFLICT
INSERT INTO counters(id, hits) VALUES (1, 1)
ON CONFLICT (id) DO UPDATE SET hits = counters.hits + 1;
RETURNING is a favourite

It lets you fetch auto-generated ids or updated values without a second round-trip — a Postgres feature interviewers like to hear about.

04JSONB

JSONB stores JSON in a decomposed binary form — slower to write than JSON but far faster to query and, crucially, indexable.

sql
SELECT data->>'email' FROM users WHERE data @> '{"active": true}';
CREATE INDEX idx_users_data ON users USING GIN (data);   -- index JSONB

Operators: -> (get JSON), ->> (get as text), @> (contains). Index with GIN for containment queries.

05CTEs & Window Functions

Postgres has excellent support for CTEs (including recursive ones) and window functions.

sql
WITH RECURSIVE subordinates AS (
  SELECT id, manager_id FROM employees WHERE id = 1
  UNION ALL
  SELECT e.id, e.manager_id FROM employees e
  JOIN subordinates s ON e.manager_id = s.id
)
SELECT * FROM subordinates;
PART 3

Indexes & Performance

06Index Types

Postgres offers many index types beyond B-tree — matching the index to the data is a strong answer.

TypeBest for
B-tree (default)Equality & range on scalar columns.
GINMulti-value columns: JSONB, arrays, full-text search.
GiSTGeometric / range / nearest-neighbour (used by PostGIS).
BRINHuge, naturally ordered tables (e.g. time-series) — tiny index.

Also: partial indexes (WHERE active) and expression indexes (LOWER(email)) to index exactly what you query.

07EXPLAIN ANALYZE

sql
EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_id = 42;

EXPLAIN shows the planned plan; ANALYZE actually runs it and reports real times and row counts. Look for Seq Scan on large filtered tables (add an index), and compare estimated vs actual rows (run ANALYZE to refresh statistics if they diverge).

PART 4

MVCC & Transactions

08MVCC & Isolation

Postgres implements MVCC: each write creates a new row version rather than overwriting, so readers never block writers and each transaction sees a consistent snapshot.

Default isolation is READ COMMITTED; REPEATABLE READ (true snapshot) and SERIALIZABLE (with serializable snapshot isolation) are available.

09VACUUM & Bloat

Because MVCC leaves dead tuples (old row versions) behind, Postgres needs VACUUM to reclaim that space and keep statistics fresh. Autovacuum runs it automatically.

Postgres-specific answer

Table bloat (from unvacuumed dead tuples) is a classic Postgres issue. Ensure autovacuum is tuned; VACUUM FULL rewrites the table to reclaim space but takes an exclusive lock. This MVCC-and-VACUUM story is what distinguishes Postgres knowledge in an interview.

PART 5

Advanced

10Extensions & Partitioning

  • Extensions make Postgres pluggable: PostGIS (geospatial), pg_stat_statements (query stats), uuid-ossp, pg_trgm (fuzzy search). Enable with CREATE EXTENSION.
  • Declarative partitioning splits a big table by range/list/hash for manageability and pruning (e.g. one partition per month of events).
PART 6

Quick-Fire Q&A

Tap a question to reveal the answer.

>Why is PostgreSQL popular?

Standards compliance, correctness, extensibility (extensions, custom types/index methods), rich types like JSONB and arrays, and strong SQL features.

>JSON vs JSONB?

JSON stores text as-is; JSONB stores a decomposed binary form — slower writes but far faster, indexable queries. Prefer JSONB.

>What is MVCC in Postgres?

Multi-version concurrency control: writes create new row versions, so readers see a snapshot and never block writers.

>What is VACUUM and why is it needed?

MVCC leaves dead tuples; VACUUM reclaims that space and refreshes statistics. Autovacuum does it automatically; neglect causes bloat.

>Default isolation level?

READ COMMITTED. REPEATABLE READ and true SERIALIZABLE (SSI) are also available.

>When would you use a GIN index?

For multi-value columns — JSONB containment, arrays, and full-text search.

>What does RETURNING do?

Returns values (like generated ids) from an INSERT/UPDATE/DELETE in the same statement, avoiding a second query.

>How do you upsert in Postgres?

INSERT ... ON CONFLICT (col) DO UPDATE/DO NOTHING.

>SERIAL vs GENERATED AS IDENTITY?

Both auto-generate keys; IDENTITY is the SQL-standard, preferred modern approach with cleaner ownership semantics.

>Why use a connection pooler with Postgres?

Postgres forks a process per connection, which is heavy; PgBouncer multiplexes many clients over fewer backend connections.

Final tips for the interview

  1. Emphasise MVCC and VACUUM — the defining Postgres topic.
  2. Prefer JSONB + GIN and explain why over plain JSON.
  3. Match the index type (GIN/GiST/BRIN) to the data.
  4. Use RETURNING and ON CONFLICT in query answers.
  5. Read EXPLAIN ANALYZE and watch estimated vs actual rows.
  6. Mention connection pooling given the per-connection process model.
