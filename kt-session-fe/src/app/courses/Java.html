PART 1

Core Concepts

01What is Kafka?

Kafka is a distributed event-streaming platform used to publish, store, and process streams of records in real time. Think of it as a super-fast, durable, distributed message queue / commit log: producers write messages, Kafka stores them, and consumers read them — all at massive scale.

Why companies use it

  • Decouples systems — the producer doesn't need to know the consumer.
  • Handles millions of messages per second.
  • Stores data durably — messages aren't deleted after reading, unlike a normal queue.
  • Multiple consumers can read the same data independently.

Common use cases: log aggregation, event sourcing, real-time analytics, microservice communication, tracking user activity, streaming ETL.

03Topics, Partitions & Offsets

A topic is a logical stream of messages. Internally it's divided into partitions, and each message gets an offset — its position number within a partition.

layout
Topic: "orders"

Partition 0:  [msg0][msg1][msg2][msg3]   ->  offsets 0,1,2,3
Partition 1:  [msg0][msg1][msg2]         ->  offsets 0,1,2
Partition 2:  [msg0][msg1]               ->  offsets 0,1

Key points to say in an interview

  • Ordering is guaranteed only within a partition, not across the whole topic.
  • Partitions allow parallelism: more partitions means more consumers can read in parallel.
  • Messages with the same key always land in the same partition, keeping per-key ordering (e.g. all events for customerId=101 stay in order).

How Kafka decides the partition

  1. If a partition is specified → use it.
  2. Else if a key is given → hash(key) % numberOfPartitions.
  3. Else → round-robin (sticky partitioning in newer versions).

04Producers — Important Configs

A producer sends messages to a topic. The most-asked producer topic is acks — the delivery guarantee.

acks valueMeaningGuarantee
acks=0Fire and forget. Don't wait for ack.Fastest, can lose data.
acks=1Wait for the leader to write.Medium. Can lose data if the leader crashes before replication.
acks=all (-1)Wait for leader + all in-sync replicas.Strongest, no data loss.

Other key producer configs

  • retries → how many times to retry on failure.
  • enable.idempotence=true → prevents duplicate messages on retry (exactly-once at the producer side).
  • batch.size & linger.ms → batch messages for throughput.
  • compression.type (snappy, gzip, lz4) → reduce network usage.

05Consumers & Consumer Groups

A consumer group is how Kafka scales reading and shares load.

The Rule

Within a consumer group, each partition is read by exactly one consumer.

example
Topic "orders" has 3 partitions (P0, P1, P2)

Consumer Group "order-service" with 3 consumers:
  Consumer A -> P0
  Consumer B -> P1
  Consumer C -> P2      (perfectly balanced)

Add a 4th consumer -> it stays IDLE (no free partition).
A consumer dies     -> its partition is reassigned (rebalancing).
Interview gold
  • Useful consumers in a group ≤ number of partitions.
  • Different consumer groups each get their own copy of all messages — two teams can read the same topic independently.
  • Rebalancing redistributes partitions when consumers join/leave. Consumption pauses briefly during a rebalance (a known pain point).

06Offset Management & Commits

The offset tells Kafka where a consumer has read up to. Consumers commit offsets so that after a restart they resume from the right place.

Commit strategies

  • Auto commit (enable.auto.commit=true) — committed automatically at intervals. Easy but risky: may commit before processing (data loss) or reprocess (duplicates).
  • Manual commit — you commit after successfully processing. Safer, more control.
    • commitSync() → blocking, reliable.
    • commitAsync() → non-blocking, faster, no retry on failure.

07Delivery Semantics

SemanticMeaningHow to achieve
At most onceMay be lost, never duplicated.Commit offset before processing.
At least onceNever lost, but may be duplicated.Commit offset after processing (common default).
Exactly onceNo loss, no duplicates.Idempotent producer + transactions (transactional.id) + read-process-write.
In practice

Most real systems use at-least-once and make the consumer idempotent — safe to process the same message twice, e.g. using a unique message ID or an upsert.

08Replication, Leader / Follower & ISR

For fault tolerance, each partition is replicated across brokers.

  • Leader — handles all reads/writes for that partition.
  • Followers — copy data from the leader.
  • ISR (In-Sync Replicas) — replicas fully caught up with the leader.
  • Replication factor = 3 is standard (1 leader + 2 followers). Survives failure of up to 2 brokers.

If the leader dies, one of the ISR followers becomes the new leader automatically.

Durability combo

min.insync.replicas is the minimum replicas that must acknowledge a write when acks=all. The classic durable setup: replication.factor=3, min.insync.replicas=2, acks=all.

09Retention & Log Compaction

Unlike a normal queue, Kafka keeps messages even after they are read.

  • Time-based retention: retention.ms (e.g. keep 7 days).
  • Size-based retention: retention.bytes.
  • Log compaction (cleanup.policy=compact): keeps only the latest value for each key. Great for "current state" topics, e.g. the latest profile per user.

10Zookeeper vs KRaft

  • Older Kafka used Zookeeper for metadata, leader election, and config.
  • Newer Kafka (2.8+, default in 3.x/4.x) uses KRaft (Kafka Raft) — Zookeeper is removed and Kafka manages its own metadata. Simpler, faster, fewer moving parts.
If asked "Is Zookeeper still needed?"

"New versions use KRaft mode and don't need Zookeeper, but many production clusters still run Zookeeper."

11Kafka Ecosystem

  • Kafka Streams — Java library to process/transform streams (map, filter, join, aggregate) inside your app.
  • Kafka Connect — ready-made connectors to move data in/out (DB → Kafka, Kafka → S3) without writing code.
  • Schema Registry — stores Avro/Protobuf/JSON schemas so producers and consumers agree on message format and can evolve safely.
PART 2

Kafka with Spring Boot

Spring Boot makes Kafka easy via spring-kafka. You mainly work with two things: KafkaTemplate to send messages, and @KafkaListener to receive them.

01Dependency & Configuration

Maven dependency

xml
<dependency>
    <groupId>org.springframework.kafka</groupId>
    <artifactId>spring-kafka</artifactId>
</dependency>

application.yml

yaml
spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.springframework.kafka.support.serializer.JsonSerializer
      acks: all
    consumer:
      group-id: order-service
      auto-offset-reset: earliest      # start from beginning if no committed offset
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.springframework.kafka.support.serializer.JsonDeserializer
      properties:
        spring.json.trusted.packages: "*"
auto-offset-reset

earliest reads all existing messages; latest reads only new ones.

02Producer with KafkaTemplate

java
@Service
public class OrderProducer {

    private final KafkaTemplate<String, Order> kafkaTemplate;

    public OrderProducer(KafkaTemplate<String, Order> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrder(Order order) {
        // key = customerId keeps same-customer messages ordered
        kafkaTemplate.send("orders", order.getCustomerId(), order);
    }
}

Async result handling (success / failure)

java
kafkaTemplate.send("orders", order.getCustomerId(), order)
    .whenComplete((result, ex) -> {
        if (ex == null) {
            System.out.println("Sent to partition " +
                result.getRecordMetadata().partition());
        } else {
            System.out.println("Failed: " + ex.getMessage());
        }
    });

03Consumer with @KafkaListener

java
@Service
public class OrderConsumer {

    @KafkaListener(topics = "orders", groupId = "order-service")
    public void consume(Order order) {
        System.out.println("Received order: " + order.getId());
        // ... process the order (save to DB, call service, etc.)
    }
}

Getting full metadata (offset, partition, key)

java
@KafkaListener(topics = "orders", groupId = "order-service")
public void consume(ConsumerRecord<String, Order> record) {
    System.out.println("Key: "       + record.key());
    System.out.println("Value: "     + record.value());
    System.out.println("Partition: " + record.partition());
    System.out.println("Offset: "    + record.offset());
}

04Manual Acknowledgment (safer processing)

Set the container to manual mode, then ack only after successful processing.

yaml
spring:
  kafka:
    listener:
      ack-mode: manual
java
@KafkaListener(topics = "orders", groupId = "order-service")
public void consume(Order order, Acknowledgment ack) {
    try {
        processOrder(order);   // do the work
        ack.acknowledge();     // commit offset ONLY after success
    } catch (Exception e) {
        // don't ack -> message will be re-delivered
        log.error("Processing failed, will retry", e);
    }
}

05Error Handling + Retry + Dead Letter Topic

This shows senior-level knowledge. Spring can auto-retry, and if it keeps failing, push the message to a Dead Letter Topic (DLT) so it's not lost.

java
@Configuration
public class KafkaErrorConfig {

    @Bean
    public DefaultErrorHandler errorHandler(KafkaTemplate<String, Object> template) {
        // On repeated failure, send to <topic>.DLT
        DeadLetterPublishingRecoverer recoverer =
                new DeadLetterPublishingRecoverer(template);

        // Retry 3 times, 1 second apart, then send to DLT
        FixedBackOff backOff = new FixedBackOff(1000L, 3L);
        return new DefaultErrorHandler(recoverer, backOff);
    }
}

Consume from the Dead Letter Topic

java
@KafkaListener(topics = "orders.DLT", groupId = "dlt-group")
public void consumeDlt(Order failedOrder) {
    log.warn("Order landed in DLT: {}", failedOrder.getId());
    // alert / store for manual inspection
}

Simpler declarative version with @RetryableTopic

java
@RetryableTopic(attempts = "4", backoff = @Backoff(delay = 2000))
@KafkaListener(topics = "orders", groupId = "order-service")
public void consume(Order order) {
    processOrder(order); // auto retries, then DLT created automatically
}

06Creating Topics from Spring Boot

java
@Configuration
public class TopicConfig {

    @Bean
    public NewTopic ordersTopic() {
        return TopicBuilder.name("orders")
                .partitions(3)
                .replicas(3)
                .build();
    }
}

Run it: docker compose up -d

Step 2Useful CLI Commands

bash
# Create a topic
kafka-topics.sh --create --topic orders --partitions 3 \
  --replication-factor 1 --bootstrap-server localhost:9092

# List topics
kafka-topics.sh --list --bootstrap-server localhost:9092

# Describe a topic (partitions, leader, ISR)
kafka-topics.sh --describe --topic orders --bootstrap-server localhost:9092

# Produce messages from console
kafka-console-producer.sh --topic orders --bootstrap-server localhost:9092

# Consume messages from console
kafka-console-consumer.sh --topic orders --from-beginning \
  --bootstrap-server localhost:9092

# Check consumer group lag (very common in production!)
kafka-consumer-groups.sh --describe --group order-service \
  --bootstrap-server localhost:9092
Consumer lag

Lag = latest offset − committed offset — how far behind a consumer is. High lag means the consumer is too slow or stuck. Interviewers love this metric.

Step 3End-to-End Flow in a Real Project

[Order API] --produce--> [Kafka topic "orders"] --consume--> [Inventory Service] | +--consume--> [Notification Service] | +--consume--> [Analytics Service]

One event, many independent consumers — this is the power of Kafka for event-driven microservices.

PART 4

Quick-Fire Q&A

Tap a question to reveal the answer — try answering out loud first.

>Difference between Kafka and a traditional message queue (RabbitMQ)?

Kafka keeps messages after they're read (log-based, replayable) and scales via partitions; multiple consumer groups can read the same data. RabbitMQ typically deletes a message once acknowledged and is a "smart broker, dumb consumer" model. Kafka is "dumb broker, smart consumer" — the consumer tracks its own offset.

>How does Kafka guarantee ordering?

Only within a partition. Use the same key for messages that must stay ordered.

>What happens if a consumer in a group fails?

A rebalance happens; its partitions are reassigned to other consumers in the group.

>Can two consumers read the same partition?

Not within the same consumer group. But consumers in different groups can.

>How do you avoid duplicate processing?

Use idempotent consumers (unique message key + upsert / dedup table), or exactly-once semantics with transactions.

>How to increase throughput?

More partitions, batching (linger.ms, batch.size), compression, and more consumers (up to partition count).

>What is consumer lag and why does it matter?

It's how many messages the consumer hasn't processed yet. Rising lag means consumers can't keep up — a key monitoring metric.

>What is the ISR?

In-Sync Replicas — replicas fully caught up with the leader; used for failover and durability guarantees with acks=all.

>Why can't you have more consumers than partitions (in one group)?

Each partition is assigned to only one consumer per group, so extra consumers stay idle.

>How do you handle a poison message (keeps failing)?

Retry with backoff, then route it to a Dead Letter Topic (DLT) for later inspection instead of blocking the partition.

>What did you use Kafka for in your projects? (prepare your own real story)

Example — "We used Kafka to decouple our order service from inventory, notification, and analytics services. Orders were keyed by customerId to preserve per-customer ordering, we used acks=all with replication factor 3 for durability, manual acknowledgment for reliable processing, and a DLT for failed messages. We monitored consumer lag in Grafana."
